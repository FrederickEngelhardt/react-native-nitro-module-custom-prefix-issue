# react-native-nitro-module prefix issues demo

This repo is to demo the issue with namespacing and autolinking with react-native and nitro modules when using prefixes that do not start with `com`

## Setup

```sh
npm install
# runs both build, codegen, and android build
npm run build-all
```

## The issue

### Description

When a Nitro module's Android Gradle `namespace` uses a non-`com.` top-level domain — such as `dev.testcompany.nitromath` — or when a *consuming app* uses a non-`com.` package name, Nitro module integration on Android breaks at **two independent layers**:

1. **Java autolinking layer**: `ReactNativeApplicationEntryPoint.java` imports a `BuildConfig` derived from the Nitro module's Gradle `namespace`, not the actual app package. When that namespace starts with `dev.*`, the generated import resolves to the wrong `BuildConfig`, causing a compile error or runtime failure.

2. **C++ JNI layer**: Nitrogen hardcodes generated Kotlin source layout under `com/margelo/nitro/` regardless of the module's Gradle `namespace`. These files are marked `DO NOT MODIFY` and the `com.margelo.nitro.` base is silently prepended with no way to suppress it.

---

### Root Cause

#### Layer 1 — Gradle Autolinking (`BuildConfig` import)

In `react-native-nitro-math`, build.gradle declares:

```groovy
android {
  namespace "dev.testcompany.nitromath"
  // ...
}
```

React Native's autolinking machinery uses this `namespace` to construct the `BuildConfig` import in the generated `ReactNativeApplicationEntryPoint.java`:

```java
// Auto-generated by RN autolinking — uses module namespace, NOT app package
import dev.testcompany.BuildConfig;  // ← derived from module's Gradle namespace
```

When a consuming app's actual package is `dev.mycompany.myapp`, its real `BuildConfig` lives at `dev.mycompany.myapp.BuildConfig`. The autolinking generator does not consult the app's configured `android.package` and instead inherits whatever namespace it discovers first from the installed Nitro modules.

**There is no mechanism in Nitrogen or the Nitro autolinking integration to override this prefix for apps whose package name does not start with `com.`.**

#### Layer 2 — Nitrogen-generated Kotlin sources (unmodifiable)

Nitrogen's nitro.json has a `"prefix"` field under android (seen in this module as `"prefix": "com"`), but this only controls the very first component. The `com.margelo.nitro.` base path is baked into the code generator. With:

```json
// nitro.json
{
  "android": {
    "prefix": "com",
    "androidNamespace": ["testcompany", "nitromath"],
    "androidCxxLibName": "NitroMath"
  }
}
```

Nitrogen generates all Kotlin sources at:

```
nitrogen/generated/android/kotlin/com/margelo/nitro/testcompany/nitromath/NitroMathOnLoad.kt
```

```kotlin
// NitroMathOnLoad.kt — DO NOT MODIFY THIS FILE
package com.margelo.nitro.testcompany.nitromath
```

Even though the module's Gradle `namespace` is `dev.testcompany.nitromath`, all generated Kotlin lives under `com.margelo.nitro.*`. Any module author wanting to use a `dev.*` source layout is blocked because generated files are always placed under `com/margelo/nitro/...`.

The `NitroMathPackage.kt` (hand-authored, at NitroMathPackage.kt) must import the generated `NitroMathOnLoad` by its Nitrogen-assigned package name, not by the Gradle namespace:

```kotlin
// NitroMathPackage.kt
import com.margelo.nitro.testcompany.nitromath.NitroMathOnLoad  // ← must use com.margelo.nitro prefix
```

There is no field in nitro.json to override the `com.margelo.nitro.` base, so the implementation class must stay under `com/margelo/nitro/...` regardless of the module's or app's naming convention.

Similarly, cpp-adapter.cpp must reference the Nitrogen-generated namespace:

```cpp
// cpp-adapter.cpp
margelo::nitro::testcompany::nitromath::registerAllNatives();
//                ^^^^^^^^^^^^^^^^^^^^ from nitro.json cxxNamespace — NOT from Gradle namespace
```

---

### Steps to Reproduce

1. Use `react-native-nitro-math` with its current build.gradle `namespace "dev.testcompany.nitromath"`.
2. Create a consuming React Native (Expo) app with Android package `dev.mycompany.myapp`.
3. Install `react-native-nitro-math` and run `yarn android`.
4. **Layer 1** — Compile failure in `ReactNativeApplicationEntryPoint.java`:
   ```
   error: package dev.testcompany does not exist
   import dev.testcompany.BuildConfig;
   ```
5. **Layer 2** — Even after patching Layer 1, any attempt to place Kotlin implementation classes under a `dev.*` path causes JNI class-not-found failures, because Nitrogen always generates `NitroMathOnLoad.kt` with `package com.margelo.nitro.testcompany.nitromath` and all references to the C++ bridge are keyed to that path.

---

### Expected Behavior

- The autolinking-generated Java file should use the **app's actual Android package** for any `BuildConfig` import, regardless of the Nitro module's Gradle `namespace`.
- Nitrogen should generate Kotlin sources at a path fully derived from the module's `androidNamespace` in nitro.json, without prepending a hardcoded `com.margelo.nitro.` base.

---

### Actual Behavior

| What is configured | What is generated |
|---|---|
| `namespace "dev.testcompany.nitromath"` in build.gradle | `import dev.testcompany.BuildConfig` in autolinking |
| `androidNamespace: ["testcompany", "nitromath"]` in nitro.json | Sources at `com/margelo/nitro/testcompany/nitromath/` (prefix hardcoded by Nitrogen) |
| Desired Kotlin at `dev/testcompany/nitromath/` | NitroMathOnLoad.kt always under `com.margelo.nitro.*` |

---

### Affected Files

- nitro.json — `"prefix": "com"` has no effect on `com.margelo.nitro.` base path
- build.gradle — `namespace "dev.testcompany.nitromath"` diverges from generated Kotlin path
- NitroMathOnLoad.kt — `DO NOT MODIFY`, always under `com.margelo.nitro.*`
- NitroMathOnLoad.cpp — C++ namespace driven by `cxxNamespace`, independent from Gradle namespace
- NitroMathPackage.kt — must hard-reference `com.margelo.nitro.testcompany.nitromath.NitroMathOnLoad`
- cpp-adapter.cpp — must use Nitrogen C++ namespace, not Gradle namespace

---

### Affected Versions

- React Native: `0.77+`
- Nitro: current (tested with `react-native-nitro-math` v0.1.0 using `dev.testcompany.nitromath` Gradle namespace)
- Android Gradle Plugin: `8.x`

---

### Workaround

Keep the module's Gradle `namespace` under `com.<something>` to match Nitrogen's hardcoded `com.margelo.nitro.` generated prefix. For the autolinking `BuildConfig` issue in consuming apps, a Gradle task via an Expo config plugin can string-replace the generated file:

```groovy
tasks.register('patchAutolinkingPackageName') {
  doLast {
    def file = file("${project.buildDir}/generated/autolinking/src/main/java/com/facebook/react/ReactNativeApplicationEntryPoint.java")
    if (file.exists()) {
      file.text = file.text.replace(
        "dev.testcompany.BuildConfig",
        "dev.mycompany.myapp.BuildConfig"
      )
    }
  }
}
afterEvaluate {
  tasks.matching { it.name.contains('compileDebugJavaWithJavac') || it.name.contains('compileReleaseJavaWithJavac') }
       .configureEach { dependsOn patchAutolinkingPackageName }
}
```

There is **no clean workaround for Layer 2**. The generated Kotlin files are `DO NOT MODIFY`, so the implementation class source path must remain under `com/margelo/nitro/...` regardless of your Gradle namespace.

---

### Requested Fix

One or more of:

1. **Autolinking should use the app's `android.package`** for any `BuildConfig` import in generated files, not the first Nitro module namespace it resolves.
2. **nitro.json's `"prefix"` field should control the full base package**, not just the TLD component, so `"prefix": "dev.testcompany"` generates sources at `dev/testcompany/nitromath/` instead of `com/margelo/nitro/testcompany/nitromath/`.
3. **Documentation** clarifying that modules with `dev.*`, `llc.*`, or any non-`com.*` Gradle namespaces require manual patching at both layers.

#### Architecture Fix (preferred long-term solution)

Remove the hardcoded `com.margelo.nitro.` prefix entirely. Treat `androidNamespace` as the **full** package path. For this module, the proposed nitro.json would become:

```json
{
  "$schema": "https://nitro.margelo.com/nitro.schema.json",
  "ios": {
    "iosModuleName": "NitroMath"
  },
  "cxxNamespace": ["dev", "testcompany", "nitromath"],
  "android": {
    "androidNamespace": ["dev", "testcompany", "nitromath"],
    "androidCxxLibName": "NitroMath"
  }
}
```

This would generate:
- Kotlin sources under `nitrogen/generated/android/kotlin/dev/testcompany/nitromath/`
- `package dev.testcompany.nitromath` in `NitroMathOnLoad.kt`
- C++ namespace `dev::testcompany::nitromath` in NitroMathOnLoad.cpp
- Gradle `namespace "dev.testcompany.nitromath"` would match the generated source layout

Currently `androidNamespace: ["testcompany", "nitromath"]` with `"prefix": "com"` produces `com.margelo.nitro.testcompany.nitromath` — the `com.margelo.nitro.` base is silently prepended with no way to suppress or replace it, creating a permanent split between the Gradle namespace and the generated source tree.
